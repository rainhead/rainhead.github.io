Status.
  • At its current stage, this document is for externalizing some of my beliefs, and organizing the work of making them more precise.
  • Many of the terms below are vague or ambiguous, please bear with me - or ask for clarification.
  • One of the best things to do next is to list positive examples of these things.
Definitions.
  • We're concerned here with specific, situated information systems. Think of a website, not a framework; a Slack channel, not Slack.
  • We understand records as facts from relational databases: atomic descriptions of the world.
  • Design is an activity where we create new language for future expression.
  • Building is an activity where we give life to some language.
Ends.
  • People with minimal training should be able to make tools for collaborative expression.
  • Designing an information system should be the same as building it.
  • It should be easy to adapt tools to local values and requirements.
  • To liberate ordinary people and organizations from their IT departments.
  • To free computer scientists from "plumbing" work.
  • Collaboration and experimentation should be possible in a "live" enrivonment.
Means.
  • Our tools should help us think about the consequences of design choices.
  • We should be comfortable making mistakes (forgiveness).
  • Our tools should accomodate subtle variations ('special cases').
  • Our tools should sit with us as we resolve conflicts.
  • The toolbox should be big enough that the purpose and power of any tool is clear, but small enough that it can be mastered in one's spare time.
Principles.
  • Information systems represent the world, and are also part of it. We think and talk through them, but we also act.
  • How information propagates through a system is a core design domain.
  • All ground truth is human speech. From the perspective of any system, they are observations of the outside world.
  • Context is a rich resource, honor it.
  • Embrace the ambiguity of new ideas, new forms: let them take names when they are ready.
  • Embrace the rigor of well-established forms.
  • Reflect an intuitive and causal sense of history.
  • Every system is open. Retain these principles when composing systems.
  • All records are immutable, and typically refer to other records.
Cases.
  • UI patterns should be first-class objects, and should allow for lightweight adaptation.
  • Because we retain input in its entirety, it should be possible to backfill any new analytic.
  • Renaming a model object should be trivial.
  • Disambiguating two cases of some model should also be as easy as possible.
  • Model user flows to reason about possible histories and requests.
  • Model a user flow before writing any implicated web pages.
Strategy.
  • Make it okay to be wrong a lot.
  • Find cheap ways to validate ideas within the actually existing world.
  • Create value early. Where successful, these ideas will allow us to provide more value faster and more cheaply than others.
  • Economize our commitments to the outside world. Stay nimble, and invest in things that will continue to teach us as our ideas evolve.
